# Constitutional Model

## Root Worldview

Wayfinder is a constitutional continuity platform. Its root obligation is to
preserve reality before interpretation, maintain continuity across changing
implementations, and keep capabilities navigable over time.

The constitutional hierarchy is:

```text
Eisengarten
-> Constitution
-> Architecture
-> Protocols and Contracts
-> Runtime
-> Implementation
```

## Constitutional Laws

| Law | Canonical force | Protected boundary |
| --- | --- | --- |
| Reality First | Reality precedes representation, interpretation, and conclusion. | Observations, evidence, and provenance cannot be replaced by derived meaning. |
| Law of Theseus | Capabilities and continuity outlive implementation parts. | Implementations may change only when identity, objective, and capability continuity are preserved. |
| One Concept, One Home | Every concept has exactly one canonical owner. | Aliases may point to one owner; duplicate definitions are constitutional drift. |
| Closed Under Extension | New behavior should compose or extend rather than mutate protected cores. | Stable contracts and capabilities must not be casually rewritten. |
| Capability First | Stable outcomes matter more than provider or implementation names. | Architecture depends on capabilities, not concrete tools. |
| Service First | Shared infrastructure belongs in services; unique behavior belongs in engines. | Engines must not secretly own reusable infrastructure. |
| Ephemeral First | Working state is disposable unless intentionally promoted. | Temporary outputs are not durable knowledge. |
| Proof Before Promotion | Durable state requires evidence and validation. | Promotion without proof is prohibited. |

## Ownership Bands

| Band | Owns | Does not own |
| --- | --- | --- |
| Philosophy | First principles, purpose, continuity obligations. | File layout, APIs, runtime mechanics. |
| Constitution | Laws, invariants, canonical stack, protected boundaries. | Implementation choices. |
| Architecture | Layer order, dependency direction, responsibility ownership. | Provider-specific logic. |
| Protocol | Contracts, schemas, capability grammar, stable interfaces. | Engine internals. |
| Runtime | Execution flow, preservation, proofs, promotion mechanics. | Constitutional meaning. |
| Implementation | Concrete modules, adapters, tools, migrations. | Canonical definitions. |

## What Must Never Change

- Reality remains prior to representation, interpretation, knowledge, and
  action.
- Reality and raw evidence are append-only unless a constitutional source
  explicitly permits a narrower lifecycle operation.
- Derived artifacts remain rebuildable and subordinate to preserved sources.
- Each concept has one canonical owner.
- Dependencies flow from later layers to earlier layers, never upstream.
- Actions affect reality and must be re-observed.
- AI and probabilistic systems must not own preservation, identity,
  normalization, or constitutional truth.
- Implementations may be replaced only by preserving capabilities and
  continuity.

## What May Evolve

- Concrete service and engine implementations.
- Provider adapters and compatibility shims.
- Derived representations, read models, indexes, and views.
- Runtime proofs, validation mechanisms, and operational maturity.
- Repository topology when migration preserves provenance and ownership.
- Canonical vocabulary through explicit governance and alias records.
